How "Automated" Actually Works

Two Interviews, Then Real Integrations — Not Magic

Once a skill exists, it runs because Claude Code wrote an actual connection to your tools. Here's the mechanism.

Discovery — ~20 min

Two interviews

You tell Claude which tools you actually use — Clio, GHL, none of the above — and what you want automated. Nothing gets built yet. This produces config files.

Build — minutes each

Real integrations

Claude Code writes an actual script that connects to your tools' APIs using credentials you provide — then runs it on-demand or on a schedule.

What "automated" means for each stage type

●

Runs on your command — you type a slash command (e.g. /new-matter), Claude Code executes the script immediately, calling your CRM's API directly.

●

Runs on a schedule — for things like the daily deadline brief, Claude Code sets up a scheduled job (cron, or a Vercel scheduled function) that runs without you touching anything.

●

Runs on a trigger — for things like "new matter created → make folders," your CRM sends a webhook the moment it happens, and a small always-on listener catches it and acts.

💡 Every one of these needs your actual API credentials (Clio key, Drive access, etc.) to talk to your real tools. The interviews tell Claude which tools to connect to — you provide the keys when you build each skill, and Claude walks you through getting them if you don't have them yet.

⚙️ Step 2 of 2 — Build

The Pipeline Runs Itself

This isn't a set of commands you trigger manually. It's one always-on service. A lead comes in, it runs end to end on its own — pausing only where a human has to act.

💡 This page assumes both interviews are done. The build below reads your config files — so it knows your state, your tools, and your own scoring rules. It doesn't work without the interviews first.
● Runs automatically
● Pauses for a human
● Off the service entirely
1

New lead arrives — triggers the service automatically

Your existing form/GHL/phone intake fires a webhook the instant a lead comes in. The service wakes up and starts working — no one clicks anything.

2

Lead scored ● RUNS AUTOMATICALLY

The moment the webhook lands, the service applies your accept/reject rules from pipeline.config.md and decides accept / reject / borderline — in under a second, with no one watching.

3

Response drafted, then it stops ● PAUSES FOR A HUMAN

The service drafts the response and sends it to you (email, Slack, wherever you choose) for approval. It does not continue on its own from here — this is the one deliberate stop in the chain. You approve or edit, then it resumes.

4

Call booked ● RUNS AUTOMATICALLY

The instant you approve the message in Stage 3, the service sends it and books the evaluation call automatically — using whatever booking tool your pipeline config says you use.

5

Case evaluation call ● OFF THE SERVICE

The service goes quiet here. Attorney meets the prospect, evaluates, decides whether to offer a retainer — entirely outside any automation. It optionally prepped talking points for you beforehand if you asked for that in the interview.

↳ Retainer offered

continues to Stage 6

↳ Case rejected here

attorney marks it declined — service drafts a closing note, no file is opened, chain ends

6

Retainer + payment ● PAUSES FOR A HUMAN — TWICE

The instant the attorney marks the case as retained, the service drafts the retainer agreement and payment request — then stops. Pause 1: a human reviews and approves the email before it sends to the client. Pause 2: once the client pays and signs, a human confirms payment actually cleared before tagging it approved. The service never sends a binding agreement or moves money on its own — both checkpoints require a person to act.

7

File opened in CRM ● RUNS AUTOMATICALLY

The instant you tag payment approved in Stage 6, the service wakes back up, creates the matter, the folders, and the matter-type-specific checklist, with zero manual step.

8

Onboarding call ● MOSTLY OFF THE SERVICE

The service can create a reminder task the moment the file opens, if you asked for that — the call itself stays entirely human.

✅ That's the whole chain, branches included. One lead, one continuous service, three human checkpoints — approve the prospect message, approve the retainer before it sends, confirm payment cleared before the file opens. Below is the single build for this.

The full pipeline — one build

Copy this into Claude Code once. It builds the whole automated chain — scoring, drafting, two approval pauses, and file-opening — as one deployed service, with the rejection branch handled too.

Read CLAUDE.md, jurisdiction.config.md, pipeline.config.md, and case-management.config.md. Build the full lead-to-file-opened pipeline as one always-on service, not a set of manual slash commands. It needs to react to events on its own, with no one triggering it by hand. It must do the following, each step firing automatically unless marked as a checkpoint: 1. LISTEN for new-lead webhooks from our intake tool (recorded in pipeline.config.md Stage 1). When one arrives, proceed immediately. 2. SCORE the lead using my exact accept/reject rules from pipeline.config.md Stage 2 — do not invent generic criteria. Output: accept / reject / borderline with the specific reason. 3. CHECKPOINT — DRAFT a response in the tone and channel from Stage 3, then STOP and send it to me for approval (email or Slack — ask me which). Do not send anything to the prospect without my explicit approval. 4. ON MY APPROVAL: send the approved message automatically, then book the evaluation call automatically using the tool from Stage 4. 5. WAIT silently through the case evaluation call (Stage 5, fully human). When the attorney logs the outcome, branch: - IF REJECTED: draft a closing/decline note for my review, log the lead as closed, and stop here. Do not open a file. - IF RETAINER OFFERED: continue to step 6. 6. CHECKPOINT (first of two) — the instant the attorney marks the case as retained, draft the retainer agreement and payment request, then STOP and send it to me for approval before anything goes to the client. Never send a binding agreement without my explicit sign-off first. 7. ON MY APPROVAL: send the approved retainer/payment request to the client, then WAIT for the signal that they paid and signed (a webhook from our CRM, e-signature tool, or payment processor, per Stage 6). 8. CHECKPOINT (second of two) — when that signal arrives, do NOT open the file automatically. Notify me that payment appears to have cleared and ask me to confirm before proceeding — payment platforms can show provisional or failed states, so a human confirms it actually cleared. 9. ON MY CONFIRMATION: automatically open the file — create the matter in [CRM from case-management.config.md] with intake data pre-filled, create the folder structure, and attach the checklist specific to the matter type, exactly as I defined per matter type in the interview. 10. If onboarding reminders were requested in Stage 8 of the interview, create that reminder task automatically once the file opens. Deploy this as a real running service (not something I run manually) — walk me through what platform makes sense (a Vercel scheduled function with webhook endpoints, or similar) given the tools I already use. Ask me for every API credential you need before building each integration, and explain how to get any I don't have yet. Security: follow all requirements in security.config.md. This service handles real prospect and client data continuously, including payment confirmation and signed agreements — credentials, logging, and access control all apply here, not just to one-off tools.
8

Onboarding call ● MOSTLY HUMAN

Attorney welcomes the new client. Claude can create a reminder task once the file opens, if your pipeline config asked for that — never the call itself.

Team Skills

Every command from the pipeline, plus drafting and automation tools, packaged as slash commands. Anyone on your team with Claude Code can run them — no server needed.

💡 How this works: all of these live in .claude/commands/ in your firm's shared repo. Anyone who clones the repo gets every command instantly — a paralegal types /draft-petition, an attorney types /case-summary.
Document Drafting

Petitions, Letters, Discovery

Reads jurisdiction.config.md for formatting, citations, and local court rules. Claude drafts, attorney reviews, nothing files without approval.

/draft-petition — paralegal types this plus case facts, gets a first draft.

Read CLAUDE.md and jurisdiction.config.md. Build a /draft-petition command. Given case facts as arguments, draft the petition form required for this matter type in our jurisdiction (read jurisdiction.config.md for the exact form, current revision date, and CRC formatting rules — font, spacing, line numbering, local rules). Output: formatted document ready for attorney review. Always flag any field where the facts I gave are incomplete or ambiguous — never guess at case facts. Nothing produced by this command files with any court automatically. Security: follow all requirements in security.config.md.

/draft-letter — client letters, opposing counsel letters, stipulations.

Read CLAUDE.md and jurisdiction.config.md. Build a /draft-letter command. Given a letter type (client update, opposing counsel, stipulation) and the relevant facts as arguments, draft the letter in our firm's tone and format from CLAUDE.md. Output: formatted draft ready for attorney review. This command never sends anything — output only. Security: follow all requirements in security.config.md.
Automations

Runs While The Case Is Open

Scheduled or triggered workflows that run on their own once a matter exists.

/deadline-check — runs on-demand, or daily on a schedule, pulled from your CRM.

Read CLAUDE.md and case-management.config.md. Build a /deadline-check command and a daily scheduled version. It should: 1. Pull all tasks and calendar entries due in the next 7 days from [CRM name from case-management.config.md] 2. Flag anything overdue in red 3. Summarize into a short daily briefing, grouped by matter Run on-demand via /deadline-check, and optionally on a daily schedule if I want that (ask me). Security: follow all requirements in security.config.md.

Runs on a schedule. Trust account low → billing alert. Case idle 14+ days → client-update prompt.

Read CLAUDE.md and case-management.config.md. Build two read-only monitors: 1. TRUST THRESHOLD MONITOR — checks trust balances against a minimum threshold I set. This is a READ-ONLY early warning, not the system of record. Never moves money or makes accounting entries. Alert me if a balance drops below threshold so I can review it myself. 2. IDLE-CASE MONITOR — flags any matter with no activity logged in 14+ days, and drafts (not sends) a suggested client-update message for my review. Security: follow all requirements in security.config.md.

Quick Commands For The Team

⚙️ One-time technical setup

How To Implement

30 minutes, once, ever. No coding background needed — just account creation and copy-paste.

1

Create 3 free accounts

Anthropic (console.anthropic.com) — sign up → API Keys → Create Key → save it somewhere safe.

GitHub (github.com) — sign up with your firm email. Stores your tools safely.

Vercel (vercel.com) — sign up with GitHub. Makes your tools live on the internet.

2

Install Claude Code

Open Terminal (Mac: Cmd+Space → "Terminal") or Command Prompt (Windows). Paste and press Enter:

npm install -g @anthropic/claude-code

Then type claude and paste your Anthropic API key when prompted.

⚠️ If npm fails, install Node.js first at nodejs.org, then retry.
3

Create your firm folder

# Mac mkdir ~/LawFirmTools && cd ~/LawFirmTools && claude
# Windows mkdir C:\LawFirmTools && cd C:\LawFirmTools && claude
4

Run the two interviews

Go to Interviews and run the Firm Brain interview, then the Pipeline interview. This is the real start of the build.

5

Set up GitHub + Vercel deployment

Tell Claude Code: "Set up this project with GitHub version control and Vercel deployment — a main branch for production, a staging branch for testing, and branch protection requiring review before merging to main."

From then on: build → test on staging → git push origin staging → review → git push origin main → live in 60 seconds.

✅ That's the whole one-time setup. Everything after this is just: go to The Pipeline or Team Skills, copy a prompt, paste it, build.
